# AI Change Log: Phase 2 v0.4 — Integrate Streaming into App + ASR Warmup

**Change-ID:** 20260622-kimi-a9893d
**Date:** 2026-06-22
**Model:** kimi
**Scope:** feature

## Reason

Complete Phase 2 of v0.4 by wiring the existing streaming audio/transcriber stack into `App` and adding eager ASR model warmup. This reduces perceived latency by running incremental transcriptions in a background thread while recording, and preloads the ASR model so the first real transcription is not delayed by model initialization.

## Changed files

- `src/config.py`
  - Added `asr_streaming_beam_size: int = 1`
  - Added `asr_warmup_at_start: bool = True`
  - Added `streaming_enabled: bool = False`
  - Added `streaming_chunk_seconds: float = 1.0`
- `src/asr/base.py`
  - Added abstract `warmup()` method to `ASRProvider`.
- `src/asr/whisper_provider.py`
  - Implemented `warmup()` that transcribes one second of silence to force model load.
  - Added optional `beam_size` parameter to `transcribe()`.
- `src/app.py`
  - Route audio chunks to both `AudioBuffer` and `StreamingTranscriber` when streaming is enabled.
  - Create/start `StreamingTranscriber` in `start_recording()`.
  - Trigger `ASRProvider.warmup()` in a background thread when `asr_warmup_at_start` is true.
  - In `stop_recording()`, stop the streaming transcriber, concatenate final streaming results, and fall back to the legacy full-audio transcription when streaming is disabled or yields no text.
- `tests/unit/test_warmup.py` (new)
  - Verify `FasterWhisperProvider.warmup()` transcribes one second of silence.
- `tests/unit/test_asr_provider.py`
  - Added tests for explicit and default `beam_size` behavior.
- `tests/unit/test_app.py`
  - Refactored fixture to a `_build_app` context manager with `streaming` and `warmup` flags.
  - Added tests for streaming routing, streaming result usage, fallback to full transcription, non-streaming legacy flow, and warmup threading.
- `tests/integration/test_end_to_end.py`
  - Disabled `asr_warmup_at_start` in the integration fixture to avoid real background threads in integration tests.
- `docs/architecture.md`
  - Updated the data-flow description to include streaming and warmup.
  - Marked Phase 1 of v0.4 complete and Phase 2 in progress.

## Risks

- `warmup()` is now an abstract method on `ASRProvider`; any future provider must implement it.
- Streaming ASR is opt-in (`streaming_enabled=False` by default), so the stable legacy pipeline remains the default.
- Streaming results currently take precedence over the final full-audio transcription when non-empty; accuracy may differ from the final transcription.
- Background warmup and streaming threads add concurrency; unit tests use mocks/deterministic fixtures, but real usage should be monitored for thread safety and resource cleanup.
